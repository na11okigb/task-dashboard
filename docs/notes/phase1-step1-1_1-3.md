# Phase 1: 開発環境の極限最適化 — Step 1-1 〜 1-3

## このセッションで学んだこと

Phase 1 の最初の3ステップを通じて、フロントエンド開発環境の基盤を構築した。
共通するテーマは **再現性（Reproducibility）** — 同じコードがいつ・どこで・誰が実行しても同じ結果になること。

## Step 1-1: Node.jsバージョン管理 — Volta

### なぜ必要か
開発者間で Node.js のバージョンが異なると「Works on My Machine（自分の環境では動く）」問題が発生する。チーム開発で最も危険なパターン。

### 3つのツールの比較

| ツール | 管理ファイル | バージョン切替 | 特徴 |
|--------|-------------|--------------|------|
| nvm | `.nvmrc` | 手動（`nvm use`） | 最も歴史が長い。シェル設定で自動化は可能だが各開発者の設定に依存 |
| fnm | `.node-version` | シェルフック設定で自動化可能 | Rust製で高速 |
| Volta | `package.json` の `volta` フィールド | ディレクトリ移動で自動切替 | インストールするだけで自動切替が有効。追加設定不要 |

### なぜVoltaを選ぶか
- **Single Source of Truth（信頼できる唯一の情報源）**: バージョン情報が `package.json` に統合される。`.nvmrc` のような別ファイルとの二重管理が不要
- **認知負荷の最小化**: 開発者はディレクトリを移動するだけ。シェル設定も不要
- **チーム展開が容易**: `git clone` → `pnpm install` だけで環境が再現される

### LTSバージョン
- 偶数バージョン（18, 20, 22...）がLTS（Long Term Support）で30ヶ月サポート
- 業務では常にLTSを選択。理由は安定性とセキュリティパッチの保証

### Voltaの制限事項
- `volta pin` は現時点で `node` と `yarn` のみ対応。pnpmのpin非対応
- 代替手段: `package.json` の `packageManager` フィールド（Node.js Corepack の公式仕様）でpnpmバージョンを固定可能
- **学び**: ツールに固執せず、目的から逆算して手段を選ぶことが重要

## Step 1-2: パッケージマネージャ — pnpm

### なぜ必要か
ライブラリ（依存パッケージ）の管理を厳格に行い、「宣言していないものは使えない」状態を作るため。

### 幽霊依存（Phantom Dependencies）
- npmは `node_modules` をフラットに展開するため、`package.json` に宣言していないパッケージ（ライブラリの内部依存）にアクセスできてしまう
- 例: `react` だけインストールしても、reactの内部依存 `loose-envify` が `import` できてしまう
- 危険性: 依存元がバージョンアップして内部依存を変更した瞬間、コードが壊れる。原因特定が困難

### pnpmの解決方法：symlink方式
- **symlink（シンボリックリンク）** = ファイルやフォルダへのショートカット。実体は別の場所にある
- pnpmの `node_modules` 構造:
  1. パッケージの実体は `.pnpm/` フォルダに隔離される
  2. `package.json` に宣言したパッケージだけ、`node_modules/` 直下にsymlinkが作られる
  3. 宣言していないパッケージへのsymlinkは存在しないため、アクセスするとエラーになる
- ルールやドキュメントではなく、**ツールの構造レベルで制約を強制**している

### ディスク効率
- npm/yarn: 同じパッケージが複数プロジェクトにコピーされる
- pnpm: **コンテンツアドレスストア**でグローバルに1回だけ保存し、ハードリンクで参照。ディスク節約

### ロックファイル（`pnpm-lock.yaml`）
- `package.json` の `"react": "^19.0.0"` はバージョン範囲指定（19.0.0以上、20.0.0未満）
- インストールタイミングで異なるバージョンが入る可能性がある
- ロックファイルは全依存の正確なバージョンとダウンロード元を記録し、**再現性を保証**する
- **必ずGitにコミットする**

## Step 1-3: ビルドツール — Vite

### なぜ必要か
ブラウザは TypeScript を直接実行できない。ビルドツールが TypeScript → JavaScript 変換、ファイルのバンドル、最適化を行う。ビルドツールの速度は開発体験（DX）に直結する。

### Webpack vs Vite

| 観点 | Webpack | Vite |
|------|---------|------|
| 起動時 | 全ファイルを解析・バンドルしてからブラウザに渡す | バンドルしない。ESMでオンデマンド変換 |
| HMR | 依存グラフ全体を辿って影響範囲を再計算 | 変更ファイルとその直接の依存だけ処理 |
| プロジェクト規模の影響 | 大きくなるほど起動・HMRが遅くなる | 規模に関係なく高速を維持 |

### ESモジュール（ESM）がViteの高速性に寄与する仕組み
- ブラウザが `import` 文をネイティブに理解し、必要なファイルを個別にリクエストできる
- Viteは起動時にバンドルせず、ブラウザからリクエストされたファイルだけを変換して返す
- だからプロジェクトの規模に関係なく起動がほぼ一瞬

### HMR（Hot Module Replacement）
- ファイル保存時にページ全体をリロードせず、変更されたモジュールだけを差し替える
- フォームの入力内容など、他の状態は保持されたまま変更が反映される

### 開発と本番の違い
- **開発時**: ESMベース、バンドルなし、速さ最優先
- **本番ビルド**: Rollupで最適化バンドル生成（ファイル結合、minify、ツリーシェイキング）
- 本番でバンドルする理由: 数百ファイルを個別にHTTPリクエストするのはネットワーク効率が悪い

### `vite.config.ts`
- Viteの設定ファイル。開発サーバーのポート、ビルドオプション、プラグインなどを設定
- 合理的なデフォルト設定があるため、多くの場合ほとんど設定不要（Convention over Configuration）

## このセッションで学んだ原則・法則

| 原則 | 内容 | 適用例 |
|------|------|--------|
| **再現性（Reproducibility）** | 同じコードがいつ・どこで・誰が実行しても同じ結果になること | Volta、ロックファイル |
| **Single Source of Truth** | 情報の管理場所を1箇所に集約し、矛盾を構造的に防ぐ | Voltaが `package.json` にバージョン情報を統合 |
| **Convention over Configuration** | 選択肢を減らしチームの認知負荷を下げる | Viteの合理的なデフォルト |
| **認知負荷の最小化** | ツール選定は「開発者が意識すべきことを減らせるか」で判断する | Voltaの自動切替、pnpmの構造的制約 |
